# sconvert

Сервис конвертации данных.

## План действий

### 1. Инфраструктура и CI/CD

**Docker** — Окружение проекта: воспроизводимый запуск локально и на сервере
**Python** — Язык приложения и скриптов
**Streamlit** — Веб-интерфейс
**Redis** — Кэш, очереди/сессии при необходимости масштабирования
**PostgreSQL** — Постоянное хранение данных
**GitHub Actions** — CI (проверки, сборка образов) и CD (выкладка на прод)

**Сервер и домен**

- Прод-сервер: **104.128.141.40** — деплой из GitHub Actions (SSH или pull образов с registry — уточнить при реализации).
- Домен **sconvert.ru** — зарезервирован; DNS пока **не делегирован**; после делегирования привязать A-запись к указанному IP и при необходимости настроить TLS (например, Let’s Encrypt).

**Порядок работ (кратко)**

1. Описать сервисы в `docker-compose` (и при необходимости отдельные Dockerfile для приложения).
2. Зафиксировать версии Python и зависимостей (`requirements.txt` / lock).
3. Workflow Actions: линт/тесты на PR; на main (или по тегу) — сборка и деплой на **104.128.141.40**.
4. Секреты в GitHub (SSH-ключ, пароли БД, `DATABASE_URL`, и т.д.) — не коммитить в репозиторий.

### 2. Приложение конвертера

1. **Базовый функционал** — конвертация между выбранными форматами (список форматов и движки — по мере проектирования).
2. **Масштабирование** — stateless Streamlit за reverse proxy; тяжёлые задачи в фоне (очередь на Redis + воркеры при росте нагрузки).
3. **Расширяемость** — модульная структура: отдельные «страницы» Streamlit (`st.navigation` / multipage app) и пакеты конвертеров; новые форматы и сценарии подключаются без переписывания ядра.
